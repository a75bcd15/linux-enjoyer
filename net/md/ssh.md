# Secure Shell
Представляет собой криптографический сетевой протокол, который обеспечивает безопасный и зашифрованный метод управления сетевыми устройствами и доступа к удаленным серверам. Широко используется администраторами и разработчиками для обеспечения безопасного удаленного доступа, передачи файлов и удаленного выполнения команд по незащищенным сетям, например, Интернет.

**Особенности:**
- поддерживает несколько методов аутентификации;
- позволяет переадресовывать сетевые порты, чтобы пользователи безопасно туннелировали другие протоколы, такие как http, https или ftp, sftp, через зашифрованное соединение;
- безопасная передача файлов: **SCP (Secure Copy Protocol)** и **SFTP (SSH File Transfer Protocol)**.

**Общие рекомендации:**
- отключить вход с правами root;
- использовать аутентификацию на основе ключа;
- ограничить доступ по SSH к определенным IP-адресам или сетям;
- постоянное обновление ПО SSH.

# Установление соединения
Есть два распространенных способа удаленного подключения к другим операционным системам. Для Windows это протокол удаленного рабочего стола (Remote Desktop Protocol, **RDP**), а для Linux — Secure Shell (**SSH**). По умолчанию SSH - протокол поверх [**TCP**](tcp.md), который работает на порте 22.
## RDP: Подключение с Linux к Windows
Используется открытый порт 3389 и команда `rdesktop` по шаблону.
- `rdesktop [IP-адрес узла Windows] -u [имя пользователя windows] -p [пароль в windows]`

Рекомендуется: 
- отключить все 3 профиля брандамуэра;
- в Параметры > Система > Удаленный рабочий стол > Дополнительные параметры > Снять галочку «Требовать использование компьютерами аутентификации на уровне сети для подключения».
## SSH: Подключение с помощью учетной записи
- `service ssh start` запустить
- `service ssh stop` остановить
- `service ssh status` узнать статус 
- `systemctl enable ssh` автоматический запуск после перезагрузки 

---
- `ssh username@IP` подключение к другой Linux машине
- `ssh username@IP -p [номер порта]` если порт был изменен
- `ssh username@IP -i [путь к ключу]` если ключ хранится в другом месте

Для того, чтобы подключаться через root пользователя:
- нужно изменить файл `/../etc/ssh/sshd_config` (с помощью sudo прав),
- убрать `#` и изменить строку `PermitRootLogin Yes`.

Стоит упомянуть, что файл `sshd_config` описывает опции SSH сервера, а `ssh_config` - клиента. 
## SSH: Подключение с помощью ключа
Цель: подключить с Kali (клиент) на Mint (сервер).
1. Сгенерировать закрытый и открытый ключ на Kali. 
	`kali@kali:~$ ssh-keygen -t rsa -b 4096`
2. Отправить открытый ключ на Mint, ввести пароль от mint.
	`kali@kali:~$ ssh-copy-id username@IP` 
3. *Проверить наличие ключа:*
	`mint@mint:~$ cat ~/.ssh/authorized_keys`
4. В Mint задать параметры только на авторизацию по ключу в файле `/../etc/ssh/sshd_config`:
	`PubkeyAuthentication yes`; `PasswordAuthentication no`
# SCP: Передача файлов
Перед отправкой установить ssh-сервер:
- `sudo apt install openssh-server -y`

Скачивание файла с сервера:
- `scp [username@IP:/удаленный-путь] [локальный-путь-назначения]`

Передача файла на сервер:
- `scp [путь-до-локального-файла] [username@IP:/каталог-на-удаленном-узле]`
